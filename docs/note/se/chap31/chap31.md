# Chapter 31: Project Management Concepts

1. **四个 P**
    - People：一个成功项目的最重要因素。
    - Product：将被构建的软件。
    - Process：完成任务所需的框架活动和软件工程任务。
    - Project：构造产品（Product）所需的全部工作。
2. **利益相关者（Stakeholders）**
    - 高级管理人员（Senior Managers）：定义了通常对项目产生重大影响的业务问题。
    - 项目（技术）经理（Project/Technical Managers）：必须规划、激励、组织并控制从事软件工作的从业者。
    - 从业者（Practitioners）：提供设计产品或应用所需的技术技能的。
    - 客户（Customers）：指定软件设计需求，以及其它能从产品中获得边缘利益的人员者。
    - 终端用户（End-Users）：软件发布生产后与终端用户互动。
3. **团队领导者（Team Leader）—— MOI Model**
    - Motivation（动力）：通过“推或拉”来鼓励技术人员发挥最佳水平的能力。
    - Organization（组织）：能够塑造现有工艺（或发明新工艺），使最初的概念能够转化为最终产品。
    - Ideas or Innovation（想法或创新）：即使必须在特定软件产品或应用设定的范围内工作，也能鼓励人们创造并感受创造力。
4. **团队协调与沟通（Team Coordination & Communication）**
    - **正式且非人际化的程序**（Formal, impersonal approaches）：包括软件工程文档和工作产品（包括源代码）、技术备忘录、项目里程碑、进度和项目控制工具（Chapter 23）、变更请求及相关文档、错误跟踪报告以及存储库数据（Chapter 26）。
    - **正式的人际程序**（Formal, interpersonal procedures）：侧重于质量保证活动（Chapter 25），包括状态审查会议、设计和规范检查。
    - **非正式的人际程序**（Informal, interpersonal procedures）：包括用于信息传播和问题解决的小组会议，以及“需求与开发人员的协调”。
    - **电子通信**（Electronic Communication）：包括电子邮件、电子公告板、视频会议系统。
    - **人际网络**（Interpersonal Networking）：包括与团队成员及项目外可能有经验或见解的人进行非正式讨论。
5. **产品范围（The Product Scope）**
    - **上下文/背景**（Context）：待构建的软件如何融入更大的系统、产品或业务环境？由于这种上下文关系，会产生哪些约束？
    - **信息目标**（Information objectives）：软件输出哪些客户可见的数据对象？输入需要哪些数据对象？
    - **功能与性能**（Function and performance）：软件执行什么功能来将输入数据转换为输出？是否有任何特殊的性能特性需要解决？
    - 软件项目范围必须明确（无歧义），且在管理层和技术层面上都是可理解的。
6. **问题分解（Problem Decomposition）**
    - 又称为“划分（partitioning）”或“问题详述（problem elaboration）”。
    - 一旦定义了产品范围（Scope）：
        - 它会被分解为组成功能（constituent functions）
        - 它会被分解为用户可见的数据对象（user-visible data objects）
        - 或者，它会被分解为一组问题类（a set of problem classes）
    - 分解过程将一直持续，直到所有的功能或问题类都被定义清楚为止。
7. **软件过程（The Process）**
    - 一旦过程框架被确定：
        - 考虑项目的特性
        - 确定项目所需的严格程度（The Degree of Rigor）
        - 为每个软件工程活动定义任务集（Task Set，Chapter 2）
8. **项目遇到困难的情形**
    - 软件人员不了解客户的需求。
    - 产品范围定义不清。
    - 变革管理不善。
    - 所选技术发生变化。
    - 业务需求变化，或者定义不清。
    - DDL 不现实。
    - 用户们抗拒。
    - 未获得充足的赞助或经费。
    - 项目团队缺乏具备相应技能的人。
    - 管理者（和从业者）回避了最佳实践和吸取的教训。
9. **项目推进常识**
    - 从正确的地方开始（Start on the right foot）：努力去理解需要解决的问题，设定切实可行的目标和期望。
    - 保持动力（Maintain Momentum）：项目经理必须提供激励措施，将人员流动降至最低，团队应强调每项任务的质量，高层管理应尽一切可能避免干扰团队。
    - 跟踪进展（Track Process）：对于软件项目，进度会随着工作成果（例如模型、源代码、测试用例集）的生成和批准进行跟踪（通过正式技术审查），作为质量保证活动的一部分。
    - 做出明智的决定（Make Smart Decision）：本质上，项目经理和软件团队的决策应当是“保持简单”。
    - 进行教训分析（Conduct a Postmortem Analysis）：建立一个统一的机制来提取每个项目的经验教训。
10. **深入项目的本质**
    - 为什么要构建这一系统？
    - 项目要做出什么？
    - 项目什么时间要完工？
    - 项目由谁负责？
    - 组织分布在哪里？
    - 如何实现这一工作（技术上和管理上）？
    - 各种资源需要用到多少（人力、软件、工具、数据库）？
11. **批判性实践**
    - 正式的风险管理（Formal risk management）
    - 经验成本与进度估算（Empirical cost and schedule estimation）
    - 基于指标的项目管理（Metrics-based project management）
    - 赚取价值追踪（Earned value tracking）
    - 针对质量目标的缺陷跟踪（Defect tracking against quality targets）
    - 以人为本的项目管理（People aware project management）

# Chapter 27: Security Engineering

## 27.1 为什么进行安全工程

- 安全是系统完整性、可用性、可靠性和安全性的先决条件。
- 安全提供了一种机制，使系统能够保护其资产免受攻击。
- **资产（Assets）**是指对利益相关者有价值的系统资源（信息、文件、程序、存储、处理器能力）。
- **攻击（Attacks）**利用漏洞来实现未经授权的系统访问。
- 仅仅通过响应漏洞报告很难提高系统安全性；安全性必须从设计之初就融入其中。

## 27.2 分析安全需求

- **风险暴露（Exposure）**是指重建丢失的系统资产所花费的时间或成本价值。
- **威胁分析（Threat Analysis）**是确定可能损坏系统资源或使未经授权者访问资源的条件或威胁的过程。
- **控制措施（Controls）**的建立是为了规避攻击并减轻攻击带来的损害。

## 27.3 在线安全威胁

- **社交媒体**：网络通常允许用户开发能够访问其个人详细信息的应用程序。
- **移动应用**：在移动设备上运行的原生应用可能拥有与设备所有者相同的资源访问权限。
- **云计算**：由于模糊了“受信任的内部”和“不受信任的外部”之间的界限，给安全领域带来了额外的机密性和信任问题。
- **物联网（IoT）**：日常物品能够通信并报告有关其用户及其环境的上下文信息。

## 27.4 安全工程分析

1. **安全需求获取**：
    - 确定用户需要如何与系统资源进行交互。
    - 编写描述系统威胁的“攻击者故事（Abuser Stories）”。
    - 使用威胁建模和风险分析，将系统安全策略确定为非功能性需求的一部分。
    - 定位攻击模式，找出解决系统安全缺陷的方案。
2. **安全建模**：
    - 涵盖策略目标、外部接口需求、软件安全需求、操作规则及安全架构描述。
    - 为设计、编码和评审提供指导。
    - 状态模型可以帮助软件工程师确保系统允许的一系列状态转换始终始于并终于安全状态。
    - 使用形式化安全模型可以提高系统的可信度，因为正确性证明可作为系统安全案例的一部分。
3. **度量设计**：
    - 安全指标应侧重于系统的可信赖性（Dependability）、可信度 （Trustworthiness）和生存能力（Survivability）。
    - 衡量资产价值和威胁可能性。
4. **正确性检查**：
    - 软件验证活动和安全测试用例必须可追溯到系统安全案例。
    - 对审计、检查和测试用例中收集的数据进行分析和总结，形成安全案例。

## 27.5 安全保障

- 用于证明你已创建了一个安全的、能激发最终用户和利益相关者信心的产品。
- **安全案例要素**：
    - 安全主张（Security claims）。
    - 将各项主张联系起来的论据（Arguments）。
    - 支持论据的证据（评审、证明等）。

## 27.6 安全风险分析

微软创建威胁模型（Threat Model）的步骤：

1. 识别资产
2. 创建架构概览
3. 应用程序分解
4. 识别威胁
5. 记录威胁
6. 评估威胁等级

## 27.7 系统可信度

- **信任（Trust）**是指软件组件与利益相关者之间可以相互依赖的信心水平。
- **验证（Verification）**确保使用客观且可量化的技术对安全需求进行评估，并可追溯到安全案例。
- 用于证明安全案例的**证据（Evidence）**必须为所有系统利益相关者所接受并信服。
- 大多数信任度量标准是基于过去在涉及信任的情况下派生出的历史数据。

# Chapter 5: Semantic Analysis 语义分析

## 5.1 Overview

1. **语义分析阶段的任务**
    - 把变量的“定义”与其“使用”联系起来——符号表。
    - 检查程序中每一个表达式的类型是否正确——类型检查。
    - 将代码转化为一种更简单的中间表示，为后续生成机器码做准备（Chapter 7）。

## 5.2 Symbol Tables 符号表

1. **定义**
    - 符号表又称环境（Environments），将变量名（Identifiers）映射到其类型（Types）和位置（Locations）。
    - 使用 $\sigma$ 来表示符号表。
    - 符号表是一个由绑定（bindings）组成的集合，例如 $\sigma_0 = \{g \mapsto string, a \mapsto int\}$ 。
2. **作用域（Scope）**
    - 程序里的每个局部变量都有自己的“可见范围”，也就是作用域 。
    - **作用域覆盖**：如果内外作用域有同名变量，右侧（新出现的）绑定信息会覆盖左侧（旧的）绑定信息 。比如 $\sigma_3 = \sigma_2 + \{a \mapsto string\}$，即使 $\sigma_2$ 中 $a$ 是 `int`，在 $\sigma_3$ 中 a 也会被当做 `string` 。注意，对于符号表的加法， $X+Y$ 不等于 $Y+X$ 。
    - **生命周期**：当语义分析到达一个作用域的结尾时，该作用域的局部绑定必须被丢弃（例如结束时丢弃 $\sigma_3$ 回到 $\sigma_1$）。
3. **多重符号表（Multiple Symbol Tables）**
    - **Java（允许前向引用 Forward Reference）**：编译器会先扫描所有的类属性，把所有环境加起来（ $\sigma_7 = \sigma_2 + \sigma_4 + \sigma_6$ ），然后在此统一的环境 $\sigma_7$ 下编译所有模块 。
    - **ML（顺序编译）**：严格按照代码出现的顺序，后面模块的编译依赖前面已经构建的环境（例如编译模块 D 时使用的是 $\sigma_0 + \sigma_2 + \sigma_4$）。

## **5.3 符号表的底层实现**

1. **命令式风格（Imperative Style）**
    - **核心原理**：直接修改全局环境 $\sigma_1$ 直到它变成 $\sigma_2$（**破坏式更新**），用完后再撤销修改，恢复成 $\sigma_1$ 。在实现上，需要一个全局环境 $\sigma$ + 一个撤销栈（undo stack）。
    - **数据结构**：哈希表 + 外部链表
        
        ![image.png](image%201.png)
        
        - 插入机制： $\sigma_1 = \sigma + \{a \mapsto int\}$，即将 $<a,\ int>$ 插入哈希表。
        - 同名变量：当出现同名变量 $\sigma' = \sigma + \{a \mapsto \tau_2\}$ 时（假设已有 $a \mapsto \tau_1$），新绑定会被插入到链表的头部（像栈一样）：`hash(a) -> <a, τ2> -> <a, τ1>` 。
        - 撤销机制：当退出作用域时，直接移除表头元素，环境自然恢复为 `<a, τ1>` 。
2. **函数式风格（Functional Style）**
    - **核心原理：**在创建新环境时，保持旧环境不变。新环境是新增绑定与旧环境之和。新环境用完后直接切换回旧环境即可。
    - **数据结构：**二叉搜索树（BST）
        
        ![image.png](image%202.png)
        
        - 如果在一个深度为 $d$ 的位置添加新节点，只需要创建 $d$ 个新节点（从根节点到插入节点的路径），树的其他分支可以直接与旧树 **共享内存**，无需复制整棵树 。
        - 平衡树查找效率为 $\log(n)$ 。

## 5.4 Tiger 编译器的符号表实现优化

Tiger 编译器在 C 语言中采用了**命令式风格**（破坏性更新） 。

1. **字符串比较优化：Symbol 模块**
    - **痛点**：每次查找都要逐个字符计算字符串的哈希，从而找到哈希桶，并使用 `strcmp` 比较字符串，非常慢 。
    - **解法**：将每个字符串转化为唯一的 `S_symbol`（符号对象）。这个对象包含了字符串名和一个指向哈希桶下一个元素的指针 。
    - **提速**：对于哈希表，直接把 `S_symbol` 的**指针地址**本身作为哈希键；比较两个符号是否相等时，直接**比较指针地址**即可，完全不需要比对字符串内容 。
2. **作用域的标记与撤销（Scope Marking）**
    - **辅助栈**：为了记录变量入表的顺序以便撤销，Tiger 引入了辅助栈机制（Auxiliary Stack） 。
    - **具体实现**：`S_beginScope` 会向符号表压入一个特殊的无意义标记记号 `<mark>` 。在 `S_endScope` 时，编译器会不断地对符号表执行 `TAB_pop` 弹出变量，直到遇到这个 `<mark>` 为止 。
    - 在底层结构中，也可以通过一个全局变量 `top` 记录最后绑定的符号，并在每个 Binder 节点中存入 `prevtop` 指针来实现 。
3. **两个独立的命名空间（Environments）**
    
    Tiger 语言中，变量 `a` 和类型 `a` 可以同时存在且不冲突，因为编译器维护了**两个独立的环境** ：
    
    - **类型环境（Type Environment）**：存储类型标识符（如 `int`, `string` 类型以及用户自定义的类型）。它将符号映射到 `Ty_ty` 结构
    - **值环境（Value Environment）**：存储变量和函数的标识符 。它将符号映射到 `E_enventry` ，该结构会区分这是一个**变量**（包含其类型）还是一个**函数**（包含参数列表和返回值类型）。
    
    ```c
    let type a = int // 为类型 a 赋值，使用 =
    var a := 1       // 为值 a 赋值，使用 :=
    ```
    
4. **Tiger 的类型**
    - Tiger 的类型分为两类：**内置原始类型**（`int`, `string`）和 **构造类型**（记录 Record 和数组 Array）。
    - **记录（Record）的唯一性**：每一次记录类型的表达式声明都会创建一个全新的类型 。
        - **非法情况**：如果定义了两个结构一模一样的记录类型 `a` 和 `b`，它们在 Tiger 眼中是不兼容的，不能互相赋值 。
        - **合法情况**：只有当 `type b = a` 时，`b` 才是 `a` 的别名，它们才被视为同一种类型 。

## 5.5 Type Checking 类型检查

1. 语义分析过程中，会对抽象语法树进行遍历，执行类型检查，主要分为两大类：
    - **表达式检查（Type-checking expressions）**：确保表达式的操作数类型匹配 。
    - **声明检查（Type-checking declarations）**：处理变量、类型、函数的声明（注意：类型、函数可能会递归声明）。
2. **表达式检查的实现**
    - `transExp` 函数
        - 输入：待检查的表达式
        - 过程：识别表达式的类型，递归检查表达式各操作数的类型
        - 返回： `expty` 结构体，包含翻译后的表达式和该表达式的值的类型
3. **声明检查的实现**
    - 在 Tiger 中，声明仅出现在 `let` 表达式中。
    - 当 `transExp` 识别到 `let` 语句时， `transExp` 会：
        - 调用 `S_beginScope` 增加 `venv` 和 `tenv` 的作用域深度 。
        - 遍历声明列表 `decs`，对每个声明调用 `transDec` 以更新环境 。
        - 调用 `transExp` 检查 `let` 的主体部分 `body` 。
        - 调用 `S_endScope` 结束作用域并返回主体表达式的结果 。
4. **各类声明的处理**
    - **变量声明**：
        - **无约束 (`var x := exp`)**：变量 `x` 的类型直接继承自初始化表达式 `exp` 的类型 。
        - **有约束 (`var x: type-id := exp`)**：必须检查约束类型与初始化表达式类型是否兼容 。
        - **特殊规则**：如果初始化表达式类型为 `Ty_Nil`，则必须显式指明其约束类型为记录类型（`Ty_Record`） 。
    - **类型声明**：通过 `transTy` 将抽象类型 `A_ty` 转换为内部类型表示 `Ty_ty`，并存入 `tenv` 。
    - **函数声明**：
        - 从 `tenv` 中查找返回类型 `resultTy` 。
        - 调用 `makeFormalTyList` 处理形式参数列表 。
        - 将函数项（`E_FunEntry`）存入 `venv` 。
        - 进入新作用域，将形式参数作为局部变量存入 `venv`，然后检查函数体表达式 。
5. **递归声明**
    - **递归类型**
        - 对于 `type list = {first: int, rest: list}`，直接处理会导致“未定义类型”错误，因为 `list` 还没处理完 。
        - **解决方案**：先将“占位头信息”存入环境 。
        - **实现**：先使用 `Ty_Name(name, NULL)` 进入环境 。接着调用 `transTy` 处理类型主体，处理完后再将返回的实际类型填充到 `Ty_Name` 结构体的 `ty` 字段中 。
    - **非法循环（Illegal Cycles）**
    Tiger 禁止不经过记录（Record）或数组（Array）的类型循环定义 。
        - **错误示例**：`type a = b; type b = a;`。这种情况下 `actual_ty` 函数将陷入死循环 。
        - 检查器必须能够检测并拦截这种非法循环 。
    - **相互递归函数**
        
        处理 `f` 调用 `g` 且 `g` 调用 `f` 的情况需要**两遍处理（Two Passes）**：
        
        - **第一遍**：收集所有函数的头部信息（名称、参数、返回类型）并放入环境，此时不处理函数体 。
        - **第二遍**：在拥有完整函数信息的新环境中，依次处理所有函数的函数体 。

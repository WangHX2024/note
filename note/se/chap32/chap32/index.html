    Chapter 32: Process and Project Metrics¶

    32.1 为何度量过程与项目¶

    32.1.1 度量的动机：我们为什么测量？¶

    一位优秀的管理者（Good Manager）必须依赖可靠的度量数据来驾驭项目。那么，我们究竟从哪些基础出发进行度量？

    1. 两大度量基础

      • 规模（Size）：用物理规模来衡量工作量，例如代码行数（LOC）。
      • 功能（Function）：用软件提供的功能量来衡量，例如功能点（FP）。

      这两种基础衍生出三类相互关联的度量方向：

      • 过程度量（Process Metrics）：关注软件开发过程本身的效能与质量。
      • 项目度量（Project Metrics）：关注特定项目的进度、成本与风险状态。
      • 产品度量（Product Metrics）：关注软件制品本身的内在质量属性（详见第 30 章）。
      • 为什么要测量（Why Do We Measure）？

      具体来说，度量为我们提供以下五种核心能力：

      1. 评估正在进行的项目状态（Assess project status）：项目现在到底在第几个里程碑？提前了还是延误了？
      2. 追踪潜在风险（Track potential risks）：哪些模块的错误率正在上升？哪个关键路径开始滑坡？
      3. 在问题演变为危机之前发现它（Uncover problem areas before critical）：与其亡羊补牢，不如防患于未然。
      4. 调整工作流程或任务分配（Adjust workflow or tasks）：发现某个工程师任务过载？数据告诉你重新分工的时机。
      5. 评估团队控制质量的能力（Evaluate quality control ability）：团队在评审中发现了多少错误？测试覆盖率是否达标？

    32.1.2 过程度量的来源与基础¶

    对软件过程效能的测量，本质上是一种间接测量（Indirect Measurement）。

    什么叫间接测量？

    我们无法直接测量"过程好不好"这个抽象概念，就好像无法直接测量"一个厨师的厨艺"一样。我们能做的，是通过观察其输出的成果来间接推断。

    过程结果（Process Outcomes）包括：

    • 发布软件前发现的错误数量（Errors uncovered before release）
    • 最终用户报告的缺陷数量（Defects reported by end-users）
    • 交付的工作制品（Work products delivered），即生产力指标
    • 投入的人力精力（Human effort expended）
    • 消耗的日历时间（Calendar time expended）
    • 进度一致性（Schedule conformance）：计划与实际的偏差
    • 其他度量

    此外，我们还通过测量特定软件工程任务的特征，来获取过程度量数据——例如，某个代码评审活动发现了多少问题，某个测试阶段耗费了多少人时。

    32.2 过程度量¶

    32.2.1 过程度量的内涵与类型¶

    过程度量主要分为以下五大类型：

    1. 质量相关度量（Quality-related Metrics）

    • 关注工作制品和可交付成果的质量。
    • 案例：每千行代码（KLOC）的缺陷数；每次代码评审发现的 Bug 数。

    2. 生产力相关度量（Productivity-related Metrics）

    • 关注投入精力与产出制品之间的比值。
    • 案例：每人月产出的功能点（FP/person-month）；每人月完成的需求条目数。

    3. 统计软件质量保证数据（Statistical SQA Data）

    • 错误分类与分析（Error Categorization & Analysis）：把错误分成"需求错误"、"设计错误"、"编码错误"等类别，分析哪类来源最多。
    • 缺陷清除效率（Defect Removal Efficiency）：测量我们在每个阶段拦截了多少缺陷，未来单独重点讲解。

    4. 缺陷传播度量（Defect Removal / Propagation）

    • 关注缺陷在开发各阶段之间如何"流动"和"扩散"。一个需求阶段引入的错误，如果在分析阶段没被发现，就会传播到设计、编码，最终修复成本指数级上升。

    5. 复用数据（Reuse Data）

    • 统计生产和复用的组件数量（Number of components）
    • 评估组件的可复用程度（Degree of reusability）

    此外，过程度量还从两个角度服务于项目管理：

    • 通过适时调整缩短开发进度、降低风险。
    • 通过持续评估产品质量，必要时调整技术方案。

    每个项目都应该测量三类数据：

    类型 说明 举例
    输入（Inputs） 完成工作所需的资源度量 工程师人数、工具许可证数量
    输出（Outputs） 过程产生的交付物度量 需求文档页数、代码行数、测试用例数
    结果（Results） 交付物有效性度量 缺陷密度、用户满意度评分

    32.2.2 过程度量准则¶

    在收集和使用过程度量数据时，必须遵循以下七条准则，否则度量不仅无益，甚至会适得其反：

    准则一：用常识和组织敏感性解读度量数据

    度量数据是工具，不是神谕。同样的缺陷密度，在不同项目规模、不同团队成熟度下，含义天壤之别。

    准则二：定期向收集度量的个人和团队提供反馈

    参与度量数据收集的工程师需要看到"我收集的数据被用来改善了什么"，否则他们会失去积极性。

    准则三：不要用度量来考核个人

    这是最危险的误用。一旦工程师知道 LOC 会影响绩效，他们会立刻开始写出冗余代码；一旦缺陷数影响评价，他们会不报告缺陷。度量必须与绩效考核解耦。

    准则四：与从业人员一起制定明确的目标和度量标准

    度量目标不能由管理层单方面制定，必须让工程师参与，才能确保度量标准贴近实际工作。

    准则五：永远不要用度量来威胁个人或团队

    "你们这个模块缺陷率第一，下个月要扣奖金！"这种做法会彻底摧毁团队信任和度量数据的真实性。

    准则六：指示问题的度量数据不应被视为"负面"数据

    发现问题本身是好事。"这个模块的圈复杂度超标"不是坏消息，而是改进机会。把红色指标视为"过程改进的信号灯"，而非追责的证据。

    准则七：不要对单一度量过度执着

    任何单一度量都有局限性。只盯着"代码行数"，会忽略架构质量；只看"缺陷数"，会忽略生产力。度量体系需要多维度平衡。

    32.2.3 软件过程改进（SPI）框架¶

    软件过程改进（Software Process Improvement, SPI） 是一个以度量为驱动的持续优化循环，其核心要素包括：

    1. 过程模型（Process Model）：如 CMMI、敏捷框架等，提供改进的参照体系。
    2. 改进目标（Improvement Goals）：明确"我们想把什么改进到什么程度"。
    3. 过程度量（Process Metrics）：收集数据，测量当前状态与目标的差距。
    4. 过程改进建议（Process Improvement Recommendations）：基于分析结果，提出具体的改进行动。

    这四个要素形成一个闭环：制定模型 → 设立目标 → 测量 → 改进 → 更新模型。

    32.3 项目度量¶

    32.3.1 项目度量的作用¶

    项目度量（Project Metrics）有两个核心作用：

    作用一：最小化开发进度滑坡风险

    通过必要的调整来避免延误、减轻潜在问题和风险。项目度量是项目经理的"早期预警系统"。

    作用二：持续评估产品质量

    在开发过程中随时评估质量现状，在必要时调整技术方案以提升质量。

    案例：挣值分析（Earned Value Analysis, EVA）

    这是项目度量中最经典的实践之一。通过将"计划工作量"与"实际完成工作量"和"实际花费成本"相比较，项目经理可以实时知晓：

    • CPI（成本绩效指数）：花了多少钱，办了多少事。
    • SPI（进度绩效指数）：按时完成了多少计划工作。

    当 CPI < 1，说明超支；当 SPI < 1，说明滞后。这些度量让"项目超支"和"进度滑坡"不再是猝不及防的噩耗，而是可以提前发现的趋势。

    32.3.2 度量准则与典型项目度量指标¶

    度量准则（Metrics Guidelines）

    项目度量与过程度量共享同一套使用准则（同 32.2.2 节所述），核心在于：数据是为改进而生，不是为惩罚而存在。

    典型项目度量指标（Typical Project Metrics）

    度量指标 含义 实际应用
    每任务花费的工时/时间（Effort/time per SE task） 每个软件工程活动消耗多少资源 识别哪类任务效率最低
    每评审小时发现的错误数（Errors per review hour） 评审活动的性价比 优化评审策略
    计划 vs 实际里程碑日期（Scheduled vs. actual milestones） 进度偏差的量化 预警延期风险
    变更数量及其特征（Changes and characteristics） 需求或设计的变动频率 评估需求稳定性
    各任务的工作量分布（Distribution of effort） 各阶段实际资源分配 发现资源浪费或欠投入

    32.4 规模导向度量¶

    在过程与项目度量中，我们需要一种"尺子"来衡量软件的规模——规模是生产力、工作量估算等几乎所有度量的分母。主要有两把"尺子"：

    32.4.1 基于代码行（LOC）的度量¶

    典型的规模导向度量（Size-Oriented Metrics）：

    度量 含义
    每千行代码的错误数（Errors per KLOC） 代码质量基础指标
    每千行代码的缺陷数（Defects per KLOC） 用户侧质量指标
    每行代码成本（$ per LOC） 开发成本估算
    每千行代码对应文档页数（Pages of doc per KLOC） 文档完整性指标
    每人月的错误数（Errors per person-month） 个人/团队错误产生效率
    每评审小时的错误数（Errors per review hour） 评审活动的效益
    每人月产出的代码行数（LOC per person-month） 开发生产力指标
    每文档页成本（$ per page of documentation） 文档生产成本

    32.4.2 基于功能点（FP）的度量¶

    典型的功能导向度量（Function-Oriented Metrics）：

    度量 含义
    每功能点的错误数（Errors per FP） 以功能维度衡量质量
    每功能点的缺陷数（Defects per FP） 用户侧功能质量
    每功能点成本（$ per FP） 功能维度的成本单价
    每功能点对应文档页数（Pages of doc per FP） 文档与功能的比例
    每人月产出的功能点数（FP per person-month） 以功能衡量生产力

    32.4.3 LOC 与 FP 的比较与选择¶

    LOC 与 FP 的对比表（Comparing LOC and FP）

    由 QSM 公司整理的行业代表性数据表明，不同编程语言完成相同功能点所需的 LOC 数量差异巨大：

    编程语言 每功能点对应的代码行数（参考值）
    C 约 128 行
    COBOL 约 105 行
    C++ 约 64 行
    Java 约 53 行
    Python 约 40 行
    SQL 约 12 行

    这意味着：用 Python 写 100 FP 的功能，LOC 约 4000 行；用 C 写同样功能，LOC 约 12800 行。如果用 LOC 衡量生产力，用 C 写代码的工程师看起来"效率是 Python 工程师 3 倍"，这显然是荒谬的。

    32.4.4 为什么选择功能点（Why Opt for FP）？¶

    功能点之所以在很多场景下优于 LOC，有以下四大理由：

    理由一：编程语言无关（Programming language independent）

    FP 从用户需求角度出发，测量"给用户提供了多少功能"，与用哪种语言实现无关。无论是 Java、Go 还是 Python，同等业务功能对应的 FP 数相同。

    理由二：在软件过程早期就可以使用（Countable early in the process）

    LOC 要等代码写出来才能数；FP 只需要需求文档就可以估算，可在项目启动阶段就用于早期成本估算。

    理由三：不"惩罚"精炼简洁的实现（Doesn't penalize inventive implementations）

    同样的功能，高手用 50 行 Python 实现，新手用 200 行实现。LOC 会"奖励"冗余代码，FP 只认功能，不认行数。

    理由四：更易度量复用组件的影响（Easier to measure impact of reuse）

    当一个组件被复用时，新增功能不需要重新开发，LOC 增量为零，但用户侧的功能确实增加了。FP 能更准确地捕捉这种场景的生产力贡献。

    32.5 面向对象与 WebApp 度量¶

    32.5.1 面向对象（OO）度量¶

    对于面向对象系统，常规的 LOC 或 FP 不能完整捕捉其规模与复杂性。需要引入以下特定度量：

    度量 含义与说明
    场景脚本数量（Number of scenario scripts） 即用例（Use Case）数量，反映系统功能需求规模
    支撑类数量（Number of support classes） 实现系统所需但与问题域不直接相关的工具类、辅助类等
    每关键类的平均支撑类数（Avg. support classes per key class） 核心分析类与支撑类的比值，反映架构的支撑复杂度
    子系统数量（Number of subsystems） 对终端用户可见的功能聚合粒度，反映系统的高层结构规模

    32.5.2 WebApp 项目度量¶

    Web 应用（WebApp）具有其独特性，需要专门的度量维度：

    度量 含义
    静态页面数量（Number of static Web pages） 内容固定、终端用户无法控制的页面数
    动态页面数量（Number of dynamic Web pages） 根据用户行为定制内容的页面数
    内部页面链接数（Number of internal page links） 站内页面间的超链接数，反映导航结构复杂度
    持久化数据对象数（Number of persistent data objects） 需要持久化存储的数据实体数量
    对接外部系统数（Number of external systems interfaced） WebApp 对接的第三方系统或 API 数量
    静态内容对象数（Number of static content objects） 图片、PDF 等固定内容资源数量
    动态内容对象数（Number of dynamic content objects） 随用户或时间变化的内容资源数量
    可执行函数数（Number of executable functions） 前端或服务端的可调用业务函数数量

    32.6 质量度量¶

    32.6.1 质量的四个核心维度¶

    软件质量（Software Quality）可以从以下四个维度来测量：

    1. 正确性（Correctness）

    软件按照规格说明运行的程度。

    • 度量方式：缺陷密度（每 KLOC 的缺陷数）、功能测试通过率、需求覆盖率。
    • 案例：一个电商平台的"库存扣减"逻辑，如果并发情况下会出现超卖，那它的正确性就是不足的。

    2. 可维护性（Maintainability）

    软件易于变更的程度。

    • 度量方式：平均修复时间（MTTR）、修改一个 Bug 需要改动的文件数、圈复杂度、技术债务量。
    • 案例：一个"上帝类"集中了 5000 行逻辑，任何修改都可能引入新 Bug，可维护性极差。

    3. 完整性（Integrity）

    软件能抵御外部攻击的程度（安全性）。

    • 度量方式：安全漏洞数量、渗透测试通过率、已知 CVE 漏洞修复率。
    • 案例：SQL 注入、XSS 跨站脚本——如果系统存在这类漏洞，完整性就是不足的。随着 AI 应用的普及，"Prompt 注入攻击（Prompt Injection）"已成为 LLM 应用完整性的新挑战。

    4. 可用性（Usability）

    软件易于使用的程度。

    • 度量方式：任务完成率、用户错误率、平均完成时间（学习曲线）、用户满意度评分（NPS/CSAT）。
    • 案例：苹果 iPhone 的设置界面的可用性，通常远高于传统路由器管理后台。

    🎭 概念比较与辨析：正确性 vs 完整性

    维度 正确性（Correctness） 完整性（Integrity）
    关注点 功能是否符合规格说明 是否能抵御外部攻击和未授权访问
    威胁来源 开发者的失误（Bug） 外部恶意行为者（攻击者）
    度量方式 缺陷密度、测试通过率 安全扫描、渗透测试
    典型案例 计算结果错误、功能缺失 SQL 注入漏洞、数据泄露

    32.6.2 缺陷清除效率（Defect Removal Efficiency, DRE）¶

    DRE 是什么？

    缺陷清除效率（DRE）是一个衡量软件团队在发布前拦截缺陷能力的核心质量度量指标。

    公式定义：

    \(\text{DRE} = \frac{E}{E + D}\)

    其中：

    • E（Errors before delivery）：软件交付给终端用户之前，在开发和测试阶段发现的错误数量。
    • D（Defects after delivery）：软件交付之后，由终端用户发现并报告的缺陷数量。

    直觉理解：

    DRE 衡量的是"我们在放行之前，拦下了多少问题"。DRE = 1.0 意味着"放行前发现了所有问题，用户零缺陷体验"。DRE = 0.5 意味着"一半的问题是用户替你发现的"。

    DRE 计算实战：

    案例一：优秀团队

    某银行 App 新版本上线前：

    • 测试阶段发现 95 个错误（E = 95）
    • 上线后用户报告 5 个缺陷（D = 5）

    \(\text{DRE} = \frac{95}{95 + 5} = \frac{95}{100} = 0.95\)

    解读：该团队在发布前拦截了 95% 的缺陷，表现优秀。业界通常将 DRE > 0.95 视为优秀水平。

    案例二：问题团队

    某创业公司急于上线：

    • 测试仅发现 20 个错误（E = 20）
    • 上线后用户报告了 80 个缺陷（D = 80）

    \(\text{DRE} = \frac{20}{20 + 80} = \frac{20}{100} = 0.20\)

    解读：该团队只拦截了 20% 的问题，80% 的缺陷都被用户发现。这不仅损害用户体验，修复"线上 Bug"的成本通常是"测试阶段 Bug"的数倍至数十倍。

    🎭 课堂互动问题：

    某团队 A 的 DRE = 0.85，团队 B 的 DRE = 0.99。管理层认为团队 B 明显更优秀，准备表彰团队 B、批评团队 A。请问，这个判断有没有可能存在问题？

    （参考答案：有可能。团队 A 可能在开发的是一个超大型、极复杂的系统，85% 的拦截率已属不易；团队 B 可能只维护一个功能极少的小工具。DRE 必须结合系统规模、复杂度和历史基线来解读，不能跨项目直接比较。此外，还需要确认两个团队对"错误"和"缺陷"的统计口径是否一致。这正是度量准则中"用常识和组织敏感性解读数据"的体现。）

    DRE 沿阶段分解（进阶内容）：

    DRE 不只能计算整体值，还可以分解到各个开发阶段，评估每个阶段的过滤效率：

    阶段 引入缺陷 本阶段发现 传播到下阶段
    需求 50 40 10
    设计 30 25 15（含上阶段遗留）
    编码 40 30 25
    测试 — 22 3（最终逃逸到用户处）

    此模型提示：越早发现缺陷，修复成本越低。需求阶段修复 1 个错误的成本，可能仅相当于上线后修复同类错误成本的 1/100。

    32.7 建立度量项目¶

    32.7.1 小型组织的度量实践¶

    对于资源有限的小型组织，以下几类核心度量已足够提供实用的管理支持：

    度量 说明
    请求评估排队时间（\(t_{queue}\)） 从提出变更请求到评估开始的等待时间
    评估工作量（\(W_{eval}\)） 完成评估所需的人时数
    评估完成到变更分配的时间（\(t_{eval}\)） 评估结束到实际开始执行变更的间隔
    执行变更的工作量（\(W_{change}\)） 实施变更所需的人时数
    执行变更消耗的时间（\(t_{change}\)） 变更实施持续的时间
    变更过程中发现的错误数（\(E_{change}\)） 变更实施过程中引入或发现的新错误
    变更发布后用户发现的缺陷数（\(D_{change}\)） 变更上线后由用户报告的缺陷数

    案例：一家 10 人的 SaaS 公司如何运用上述度量

    该公司将上述 7 个指标录入简单的 Excel 表格，每次版本发布后更新一次。三个月后，他们发现：

    • \(t_{queue}\) 平均长达 5 天（变更请求长期积压）
    • \(W{change}\) 是 \(W{eval}\) 的 10 倍（评估严重低估了实施复杂度）

    基于这两个发现，他们引入了轻量级的变更影响评估流程，在评估阶段邀请实施工程师共同参与，将 \(W{change}\) 与 \(W{eval}\) 的比值从 10 降至 4，同时 \(t_{queue}\) 缩短至 1.5 天。

    这说明：即使是最简单的 7 个度量数据，也能驱动有意义的过程改进。

    32.7.2 建立度量项目的十步法 Establishing a Metrics Program¶

    建立一个正式的度量项目，需要有序地完成以下十个步骤：

    Step 1：识别业务目标（Identify business goals）

    在开始任何度量之前，首先明确组织的核心业务目标是什么。一切度量活动必须服务于业务目标，而不是为度量而度量。

    • 案例：某公司目标是"18 个月内将新产品的市场份额提升 10%"。

    Step 2：识别你想知道或学习的内容（Identify what you want to know）

    从业务目标出发，明确"为了实现这个目标，我需要了解哪些信息"。

    • 案例：为了提升市场份额，需要了解"我们的产品上线速度和质量与竞争对手相比如何"。

    Step 3：识别子目标（Identify subgoals）

    将宏观目标分解为若干可测量的子目标。

    • 案例：子目标 1 = 缩短功能交付周期；子目标 2 = 降低上线后缺陷率。

    Step 4：识别相关实体与属性（Identify entities and attributes）

    明确度量的对象是什么（实体），以及要测量它的哪些方面（属性）。

    • 案例：实体 = 代码模块、迭代周期；属性 = 缺陷密度、交付时间。

    Step 5：正式化度量目标（Formalize measurement goals）

    用明确的语言和可操作的标准，将度量目标正式记录下来。推荐采用 GQM（目标-问题-度量）模式（参见第 30 章）。

    Step 6：识别可量化问题与相关指标（Identify quantifiable questions and indicators）

    将抽象的度量目标转化为具体可回答的量化问题，并确定对应的指标。

    • 案例：问题 = "我们每次迭代平均修复了多少缺陷？" → 指标 = 每迭代的 DRE。

    Step 7：识别将要收集的数据元素（Identify data elements to collect）

    明确为了构建上述指标，具体需要收集哪些原始数据。

    • 案例：需要收集"测试阶段发现的 Bug 数"和"上线后用户报告的 Bug 数"。

    Step 8：定义使用的度量，并使其操作化（Define measures and make operational）

    为每一个度量写下清晰的定义和计算规则，确保整个团队执行口径一致。

    • 案例：DRE = E / (E + D)，其中 E 定义为……D 定义为……

    Step 9：识别实施度量所需的行动（Identify actions to implement）

    明确谁在什么时候用什么工具来收集这些数据。

    • 案例：在 JIRA 中新增"Bug 来源（测试发现/用户报告）"字段，由测试工程师在关闭 Bug 时填写。

    Step 10：准备实施度量的计划（Prepare an implementation plan）

    将上述所有步骤整合为一份完整的度量实施计划，包含责任人、时间节点和回顾机制。

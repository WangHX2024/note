    Chapter 3-3: Parsing 语法分析¶

    3.11 Automatic Parser Generation¶

    1. Yacc 概述

      • 功能：语法解析器生成器

      • 输入与输出：输入是一个后缀为 .y 的规范文件，输出是一个包含 C 语言源代码的文件（名为 tab.c，实现 LALR 解析表） 。

      • 规范文件的基本结构：分为三部分，由 %% 分隔 ：

        {definitions}
%%
{rules}
%%
{auxiliary routines}
        • {definitions}：定义部分（包括 C 头文件、Token 声明等） 。
        • {rules}：语法规则部分 。
        • {auxiliary routines}：辅助例程部分（如 main 和 yyerror 等） 。
        • Yacc 的 Token 声明
          • 在定义部分对 Token 进行声明，Yacc 会自动为该 Token 类型分配一个编号，供后续使用。
          • 单字符 Token 无需声明，Yacc 已经为所有的单字符分配了编号，直接使用即可。
          • Token 声明时，暂无需指明其数据类型。
      %token NUMBER // 声明了一个名为 NUMBER 的 Token
%start symbol // 声明 symbol 为起始符号

    2. Yacc 的工作流程与辅助例程（Auxiliary Routines）

      • yyparse：主解析函数，返回整数（0 表示解析成功，1 表示失败）。
      • yylex：词法分析函数，被 yyparse 调用 。它返回 Token 的类型，并将其语义值存储在全局变量 yylval 中（若 Token 没有语义值则存储 0） 。当返回 0 时，Yacc 认为输入结束 。
      • yyerror：当解析遇到错误时被调用，用于打印错误信息 。

    3. Yacc 的语法规则（Rules）与语义动作（Actions）

      • 基本格式：Rule {Action Code} 。
      • 执行时机：当解析器使用某条规则执行规约（reduce）动作后，就会执行对应的动作代码 。

      • 伪变量（Pseudo variables）：

        • $$：代表规则左部（LHS）的语义值 。
        • $1, $2, $3...：代表规则右部（RHS）第 i 个符号的语义值。

      • 嵌入式动作（Embedded Actions）：有时需要在一条规则完全匹配之前就执行代码，Yacc 允许在规则选择的中间插入动作代码 。

        // 例：在解析形如 int a, b, c; 的变量列表前，先记录变量类型
decl: type {current_type=$1} var-list;
type: INT {$$=INT_TYPE;}
    | FLOAT {$$=FLOAT_TYPE;}
    ;
var_list: var_list ',' ID {setType(tokenString, current_type);}
        | ID {setType(tokenString, current_type);}
        ;

        嵌入式动作代码块会占据一个编号位置。例如：

        list: item1 { do_item1(**$1**); } item2 { do_item2(**$3**); } item3

        其中， $2 用来记录第一个嵌入式动作代码块的返回值。

    4. 处理不同的数据类型 (YYSTYPE 与 %union)

      • 默认情况下，所有语义值（$$, $i, yylval）的类型由宏 YYSTYPE 定义，默认为 int 。

        #define YYSTYPE int

      • 问题：在复杂的编译器中，不同的符号可能有不同的数据类型（比如有的是浮点数，有的是操作符字符，有的是 AST 节点） 。

      • 解决方案：

        • 使用 %union 在 Yacc 规范中直接声明一个 C 联合体，包含所有可能的类型 。这会在生成的代码中创建一个名为 YYSTYPE 的联合体。

          %union {
    double val; // 用于存储数值（如 3.14）
    char op;    // 用于存储操作符（如 '+'）
}

        • 使用 %type 将联合体中定义的特定类型绑定到特定的非终结符或终结符上（例如 %type <成员名> 符号名） 。

          %type <val> exp term NUMBER
// 表示 exp、term 和 NUMBER 都为 double 类型。

    5. Yacc 的冲突报告与解决

      • 移入-规约冲突：默认选择移入（shifting）。
      • 规约-规约冲突：默认使用在文法中较早出现的规则 。

    6. Yacc 的优先级指令（Precedence Directives）

      • 解决方案

        • 优先级（Operator Precedence）
          • 规则： 优先级高的运算符会促使解析器执行 移进。
          • 示例： 在处理 E -> E + E . 且下一个字符是 * 时，因为 优先级高于 +，解析器选择移进（先算乘法）。

        • 结合性（Associativity）

          当遇到相同优先级的运算符时（例如 1 + 2 + 3），需要结合性来决定：

          • 左结合： 选择 归约。例如 %left '+' 会让 1 + 2 + 3 解析为 (1 + 2) + 3。
          • 右结合： 选择 移进 (Shift)。例如 %right '=' 会让 a = b = c 解析为 a = (b = c)。
          • 非结合： 如果 a != b != c 这种写法是非法的，则使用 %nonassoc '!='。如果用户写了这样的表达式，解析器会报错。
            • 声明方式：声明的顺序决定了优先级， 写在后面的行优先级更高。
        %nonassoc EQ NEQ // 最低优先级
%left PLUS MINUS
%left TIMES DIV   
%right EXP // 最高优先级

      • 特定规则优先级（%prec）：用于一元减号（-）等情况，可以使用 %prec UMINUS 强制赋予该规则特定的最高优先级 。

      • 语法 vs. 语义（Syntax v.s. Semantics）
      • 陷阱：不要试图在语法层面区分所有细节（例如算术表达式和布尔表达式不能相加），这会导致复杂的规约-规约冲突 。
        • 语法（Syntax）负责结构的正确性（比如括号是否配对，赋值号位置是否正确）。
        • 语义（Semantics）负责类型的正确性（比如变量是否声明，类型是否匹配）。
      • 最佳实践：在语法层面保持宽泛（接受所有表达式），将类型检查推迟到编译器的“语义分析阶段” 。

    3.12 Error Recovery¶

    动机：开发者希望一次编译能报告程序中的所有错误，而不是遇到第一个错误就停止。

    1. 局部错误恢复（Local Error Recovery）

      • 局部恢复通过在检测到错误的位置调整解析栈和输入流，以便恢复解析。

      • Yacc 的 error 符号：Yacc 会对规则进行改写，引入一个特殊的预定义 Token error 来控制恢复过程 。它被视为一个终结符 。

        // 一般引入以下两条规则
exp -> (error)
exps -> error ; exp

      • 恢复步骤：

        • 遇到错误时，弹出解析栈，直到到达一个可以移入 error 符号的状态 。
        • 移入 error 符号 。
        • 丢弃接下来的输入符号，直到找到一个可以在当前状态下执行非错误动作（例如遇到分号 ; 或右括号 )）的 Lookahead 符号 。
        • 恢复正常解析 。
      • 注意事项（副作用问题）：如果与规约动作绑定的语义动作有副作用（例如影响一些临时变量），在局部错误恢复期间，错误地弹出栈会导致这些副作用无法被撤销，从而引发逻辑错误。因此，要求语义动作尽量无副作用 。
      • 全局错误修复（Global Error Repair - Burke-Fisher 方法）
      • 全局修复旨在寻找最小的插入和删除集合，将源字符串变为语法正确的字符串 。
      • 核心思想：在解析器报告错误之前的 K 个 Token 范围内，尝试所有可能的单 Token 插入、删除或替换 。如果某次修复能让解析器继续向后成功解析 R（如 4）个 Token，则认为该修复是“足够好”的 。
      • 实现机制：
        • 数据结构：
          • 当前栈（Current Stack）：记录当前解析器处理到的最新状态。
          • 旧栈（Old Stack）：记录 K 个 Token 之前的解析器状态。它是我们的“存档点”。
          • K 长度队列：存储从“旧栈”到“当前栈”之间所有的 Token。
        • 运作流程：每当解析器读入一个新的 Token：
          • 将其移入当前栈并执行相应的归约操作。
          • 将其放入队列的末尾。
          • 队列头部的 Token 被弹出，并移交给旧栈。旧栈同步进行移入和归约，并执行对应的语义动作。
          • 直白理解：旧栈总是比当前栈慢 K 个步子。如果当前位置报错了，我们还有 K 个位置的“缓冲带”可以进行修改。
        • 错误恢复触发机制：当解析器在处理当前 Token 时发现无法匹配任何语法规则，触发修复程序：首先回退到旧栈所处的状态，然后对队列中的每一个位置尝试所有可能的单 Token 插入、删除或替换 。如果某次修复能让解析器继续向后成功解析 R 个 Token，则认为该修复是“足够好”的 。
      • 仅在 Token 进入旧栈时，才执行对应的语义动作，因此没有副作用问题。

    Chapter 8: Blocks & Traces¶

    8.1 概述¶

    1. 背景

      在语义分析阶段生成的中间表示（IR）树需要被转换为汇编语言或机器语言 。虽然树语言（IR）的操作符在选择时已经尽量匹配大多数机器的功能 ，但树语言和机器语言之间仍然存在一些不匹配的方面 ：

      • 条件跳转的差异： 树语言中的 CJUMP 指令可以跳转到两个标签中的任意一个 。然而，真实机器的条件跳转指令在条件为假时，通常会直接向下执行（fall through）到下条指令 。
      • 表达式内嵌语句的困扰： 在表达式内部包含 ESEQ 节点会带来极大的不便 。 子树的不同求值顺序会导致不同的结果 。但在实际编译优化中，能够以任意顺序对表达式的子表达式进行求值是非常有用的 。
      • 内嵌函数调用的冲突： 当一个 CALL 节点位于另一个 CALL 节点的参数表达式中时（如 CALL(f, [e1, CALL(g, [e2, ...])])），在尝试将参数放入固定的一组形式参数寄存器时就会产生冲突 。
      • 解决方案

      为了消除上述不匹配问题，编译器需要分三个阶段对树进行转换 ：

      1. 生成规范树（Canonical Trees）： 将一棵树重写为由多个规范树组成的列表，这些规范树中不包含 SEQ 或 ESEQ 节点 。
      2. 生成基本块（Basic Blocks）： 将上述列表分组，构成由基本块组成的集合。基本块内部不能包含跳转指令或标签（即单入单出） 。
      3. 生成轨迹（Traces）： 将基本块排序为一个轨迹集合，在这些轨迹中，每一个 CJUMP 指令都紧跟其条件为假时的跳转标签 。

    8.2 规范树 Canonical Trees¶

    1. 规范树的定义与属性

      规范树必须严格满足以下两个属性 ：

      • 属性 1： 不包含任何 SEQ 或 ESEQ 节点 。这意味着每棵规范树只包含一个语句节点，且必须是根节点，其余所有节点均为表达式节点 。
      • 属性 2： 每个 CALL 节点的父节点只能是 \(EXP(...)\) 或 \(MOVE(TEMP~t, ...)\) 。结合属性 1 可知，CALL 节点的父节点必须是规范树的根节点 。这也推导出，一棵规范树中最多只能包含一个 CALL 节点 。
      • 转换为规范树的三个步骤

      为了完成由中间表示树到规范树的转换，需要执行以下三个操作 ：

      • 步骤 A：消除 ESEQ
      • 步骤 B：将 CALL 提升至顶层
      • 步骤 C：消除 SEQ
      • 步骤 A：消除 ESEQ（Eliminate ESEQs）
      • 核心思想： ESEQ(s, e) 的语义是先计算语句 s 以产生副作用，然后再计算表达式 e 以获取结果 。消除的思路是将 ESEQ 在树中不断向上提升，直到它的父节点是一个语句节点，此时它就可以转化为一个 SEQ 节点 。

      • 转换规则：

        • 当 ESEQ 嵌套在另一个 ESEQ 中： \(ESEQ(s_1, ESEQ(s_2, e))\) 被转换为 \(ESEQ(SEQ(s_1, s_2), e)\) 。

        • 当 ESEQ 位于二元操作符的左侧： \(BINOP(op, ESEQ(s, e_1), e_2)\) 被转换为 \(ESEQ(s, BINOP(op, e_1, e_2))\) 。该规则同样适用于 MEM、JUMP 和 CJUMP 。

        • 当 ESEQ 位于二元操作符的右侧： \(BINOP(op, e_1, ESEQ(s, e_2))\)，需要判断语句 s 和表达式 e1 是否满足 交换律（Commute）。

          关于交换律：

          • 如果 s 分配的临时变量和内存位置没有被 e1 引用，并且它们不共同执行外部 I/O，则它们可以交换 。
          • 编译器在静态编译时无法总是准确知道这一点，因此采用保守近似：常量（ \(CONST(i)\)、 \(NAME(n)\)）或空语句可以与任何元素交换；除此之外的其他情况均假设为不可交换 。

          • 如果可交换： 直接提取： \(ESEQ(s, BINOP(op, e_1, e_2))\) 。

          • 如果不可交换： 需要引入一个新的临时变量 t 来保存 e1 的值，以保证求值顺序。转换结果为： \(ESEQ(MOVE(TEMP~t, e_1), ESEQ(s, BINOP(op, TEMP~t, e_2)))\) 。

          • 步骤 B：将 CALL 移至顶层（Move CALLs to Top Level）
            • 问题： 树语言允许 CALL 作为子表达式存在，但函数返回值通常存放在同一个专用的返回值寄存器 \(TEMP(RV)\) 中 。如果在表达式中有多个调用（如 \(BINOP(PLUS, CALL(...), CALL(...))\)），第二次调用会在加法执行前覆盖掉第一次调用存在 RV 寄存器中的结果 。
            • 解决方案： 强制将每个返回值立即赋给一个全新的临时寄存器 。即将 \(CALL(fun, args)\) 重写为 \(ESEQ(MOVE(TEMP~t, CALL(fun, args)), TEMP~t)\) 。随后，消除 ESEQ 的步骤会自动将这个 MOVE 语句向上提升到包含它的表达式外部 。
          • 步骤 C：线性化列表并消除 SEQ（A Linear List of Statements）
            • 过程： 在处理完整个函数体后，所有的 SEQ 节点都会被提升到树的顶层，形成类似 \(SEQ(SEQ(SEQ(..., s_x), s_y), s_z)\) 的结构 。
            • 线性化： 通过反复应用结合律规则 \(SEQ(SEQ(a, b), c) = SEQ(a, SEQ(b, c))\)，可以将嵌套的 SEQ 展平 。
            • 最终结果： 展平后得到一个纯粹的线性语句列表 \(s_1, s_2, ..., s_{n-1}, s_n\) 。在这个最终列表中，任何一条语句 \(s_i\) 都不再包含 SEQ 或 ESEQ 节点，成功完成了规范树的转换 。

    8.3 基本块 Basic Blocks¶

    1. 概念与动机
      • 在决定程序中跳转目标的去向时，我们实际上是在分析程序的控制流（Control Flow） 。
      • 控制流是程序中指令的执行顺序 。
      • 在分析控制流时，可以忽略寄存器和内存中的数据值，以及算术计算过程 。
      • 任何非跳转指令在控制流分析中都没有特殊的行为意义 。
      • 因此，我们可以将任何非分支指令序列合并在一起，形成一个“基本块”，然后集中分析基本块之间的控制流 。

    2. 基本块的定义

      一个基本块是一系列语句的集合，它必须满足“单入单出”的特性，即总是从开头进入，在末尾退出 。具体规则如下：

      • 第一条语句必须是一个 LABEL（标签） 。
      • 最后一条语句必须是一个 JUMP（无条件跳转）或 CJUMP（条件跳转） 。
      • 在基本块的内部，不允许包含任何其他的 LABEL、JUMP 或 CJUMP 指令 。
      • 生成基本块的算法

      将一个长语句序列划分为基本块的算法步骤如下：

      • 从头到尾扫描整个语句序列 。
      • 每当遇到一个 LABEL 时，就开始一个新的基本块（同时也意味着上一个基本块的结束） 。
      • 每当遇到一个 JUMP 或 CJUMP 时，就结束当前基本块（同时也意味着下一个基本块即将开始） 。
      • 补全机制： 如果按照上述划分，某个基本块没有以 JUMP 或 CJUMP 结尾，则需要在这个块的末尾显式地追加一条跳转到下一个块标签的 JUMP 指令 。
      • 补全机制： 如果某个基本块在开头没有 LABEL，则需要发明一个新的 LABEL 并将其插入到开头 。

    8.4 轨迹 Traces¶

    1. 概念与动机
      • 基本块可以以任意顺序排列，程序执行的最终结果都是相同的 。
      • 基于这一点，我们可以选择一种特定的基本块排列顺序，以满足“每个 CJUMP 后面紧跟其条件为假（false label）的标签”这一条件 。
      • 我们还可以对块进行排列，使得许多无条件 JUMP 紧跟着它们的目标标签 。
      • 这样就可以删除这些多余的 JUMP 指令，从而使编译后的程序运行得更快 。
    2. 轨迹的定义
      • 轨迹（Trace）是程序执行期间可能被连续执行的一组语句序列，其中可以包含条件分支 。
      • 目标是找到一个能够精确覆盖整个程序的轨迹集合，即每个基本块必须且只能存在于一个轨迹中 。
      • 我们希望覆盖集合中的轨迹数量尽可能少，以最小化从一个轨迹跳转到另一个轨迹的 JUMP 数量 。

    3. 生成轨迹的算法

      寻找覆盖程序轨迹集合的核心思想是基于控制流图的深度优先搜索（Depth-First Search） ：

      • 将程序所有的基本块放入一个列表 Q 中 。
      • 当 Q 不为空时，开启一个新的空轨迹 T，并从 Q 中移除头部元素 b 。
      • 当 b 未被标记时，标记 b，并将其追加到当前轨迹 T 的末尾 。
      • 检查 b 的后继节点（即 b 分支指向的块）；如果存在任何未被标记的后继节点 c，则将 c 赋值给 b 继续循环 。
      • 如果 b 的所有后继节点都已被标记，则结束当前轨迹 T 。
      • 收尾与调整（Finishing Up）

      由上述算法生成的轨迹中，CJUMP 依然可能没有紧跟着它的 false label 。因此，需要将所有轨迹展平为一个长语句列表，并进行以下调整 ：

      • 情况 A： 如果 CJUMP 紧跟着它的 false label，则保留不动 。
      • 情况 B： 如果 CJUMP 紧跟着它的 true label，则交换 true 和 false 标签，并对判断条件取反 。
      • 情况 C： 如果 CJUMP 后面既不是 true label 也不是 false label，则发明一个新的 false 标签（例如 If'），并将这条 CJUMP 语句重写为三条语句，以强行满足条件 。这三条语句依次为：使用新标签的 CJUMP、新标签 LABEL If'、跳转到原 false 标签的 JUMP 。

    4. 最优轨迹（Optimal Traces）

      • 任何频繁执行的指令序列（例如循环体）都应该占据它自己独立的轨迹 。
      • 这样有助于最小化无条件跳转的数量 。
      • 将频繁执行的代码集中在一起也有助于其他类型的优化，例如寄存器分配和指令调度 。
